# setup_branch_protection: ブランチ保護ルールの設定

## CONTEXT

- 対象: チームリーダー（鮫島）のみが実行する
- 目的: `main` および `develop` ブランチへの直接 push を禁止し、Pull Request 経由のみマージ可能にする
- 実行環境: ブラウザ（GitHub）
- 前提条件:
  - リポジトリへの Settings 権限があること
   - `GASプロジェクトの作成.md` が完了していること

---

## STEP 1: Settings 権限の確認

### ACTION
以下のURLを開く

```
https://github.com/RUSAEXP1EMB2026/【リポジトリ名】/settings
```

`【リポジトリ名】` を対象のリポジトリ名に置き換える

### VERIFY

| 表示 | 状態 | 次のアクション |
|---|---|---|
| Settings ページが表示される | 権限あり | STEP 2 へ進む |
| 404 または「Access denied」が表示される | 権限なし | TA に Settings 権限の付与またはブランチ保護の設定を依頼する。このドキュメントをTAに共有して以降の手順を実行してもらう |

---

## STEP 2: Rulesets のページを開く

### ACTION
Settings ページの左サイドバーから以下を選択する

```
Code and automation
→ Branches
```

### VERIFY
「Branch protection rules」または「Rulesets」のセクションが表示されていること

---

## STEP 3: main ブランチの保護ルールを作成する

### ACTION

#### 3-1. 新しいルールセットを作成する

```
「Add branch ruleset」または「Add rule」をクリックする
```

#### 3-2. 以下の設定を入力する

| 項目 | 設定値 |
|---|---|
| Ruleset Name | `protect-main` |
| Enforcement status | `Active` |
| Target branches | `main` |

「Add target」→「Include by pattern」を選択し、以下を入力する

```
main
```

#### 3-3. 以下のルールを有効にする

| ルール | 設定 | 目的 |
|---|---|---|
| Restrict deletions | オン | main ブランチの削除を禁止する |
| Require a pull request before merging | オン | 直接 push を禁止し PR 経由のみにする |
| ↳ Required approvals | `0` | 承認者数（承認なしでもマージ可能にする） |
| ↳ Dismiss stale pull request approvals when new commits are pushed | オン | 新しいコミット後に承認をリセットする |
| Block force pushes | オン | 強制 push を禁止する |

#### NOTE
- `Required approvals` を `0` にすることで、承認がなくても PR を作成すればマージできる
- チームの習熟度に応じて後から `1` に変更してよい

#### 3-4. 保存する

```
「Create」または「Save changes」をクリックする
```

### VERIFY
Branches ページのルール一覧に `protect-main` が `Active` で表示されていること

## STEP 5: ブランチ保護ルールの動作を確認する

### ACTION
VS Code のターミナルで以下を実行して、直接 push が拒否されるか確認する

```bash
# main ブランチに切り替える
git checkout main

# テスト用の空コミットを作成する
git commit --allow-empty -m "test: ブランチ保護の確認"

# main に直接 push を試みる
git push origin main
```

### VERIFY

| 出力 | 状態 | 次のアクション |
|---|---|---|
| `remote: error: GH006: Protected branch update failed` などのエラーが表示される | 保護ルール正常動作 | STEP 6 へ進む |
| push が成功してしまう | 保護ルール未適用 | STEP 3 に戻り設定を確認する |

### ACTION（確認後のクリーンアップ）
テスト用コミットをリセットする

```bash
git reset HEAD~1
git checkout develop
```

## STEP 6: Pull Request の作成手順をチームに周知する

### CONTEXT
ブランチ保護により、`feature/◯◯` → `develop` へのマージは Pull Request 経由が必須になる。
チームメンバーが PR の作成方法を知っている必要があるため、以下の手順を共有する。

### PR 作成手順（チームメンバー全員への共有内容）

```
1. GitHub でリポジトリを開く
   https://github.com/RUSAEXP1EMB2026/【リポジトリ名】

2. 「Pull requests」タブをクリックする

3. 「New pull request」をクリックする

4. 以下を設定する
   base:    develop        ← マージ先（変更しない）
   compare: feature/◯◯   ← 自分の作業ブランチを選択する

5. 「Create pull request」をクリックする

6. タイトルと説明を入力する
   タイトル例: feat: aircon.js の冷房判定を実装
   説明: 変更内容の概要を日本語で記載する

7. 「Create pull request」をクリックして PR を作成する

8. チームリーダーまたはTA に PR のURLを共有してマージを依頼する
```

### ACTION
上記の手順を Discord またはチームの連絡手段でメンバーに共有する

---

## STEP 7: develop → main へのマージルールを確認する

### CONTEXT
`develop` → `main` へのマージはチームリーダーのみが行う。
以下のタイミングで実施する。

| タイミング | 条件 |
|---|---|
| 機能の節目（例: Phase 1 完成時） | develop 上で全モジュールの動作確認が完了している |
| TA・授業のレビュー前 | main に安定版を反映しておく必要がある場合 |

### ACTION
`develop` → `main` へのマージ手順をチームに周知し、必要に応じて `docs索引.md` の参照順を更新する

```markdown
## ブランチ運用ルール（周知テンプレート）

### feature → develop（各自が実施）
1. 作業が完了したら `git push origin feature/◯◯` を実行する
2. GitHub で Pull Request を作成する（base: develop）
3. チームリーダーに PR のURLを共有する

### develop → main（リーダーのみが実施）
1. GitHub で Pull Request を作成する（base: main, compare: develop）
2. 動作確認済みであることを確認してマージする
```
