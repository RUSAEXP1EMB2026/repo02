# 要求仕様書

## 全体概要

### システムの概要
スマートフォンのGPS情報を用いてユーザーの外出・在宅を自動判定する。外出時は、テレビ・照明・クーラー等の消し忘れをDiscordに通知し、リモート操作で機器をオフにできる。在宅時は、Nature Remo 3 のセンサから室温を定期取得し、論文根拠に基づく最適温度帯（22〜23℃）を維持するようクーラーを自動制御する。

### 製品の機能
スマートフォンのGPS位置情報を定期取得し、自宅から設定した距離（ジオフェンス半径）を2回連続で超えた場合に「外出」と判定する。外出判定時、監視対象機器の稼働状況を取得し、オン状態の機器があれば機器名・稼働時間・設定内容を含む通知をDiscordの指定チャンネルに送信する。ユーザーはDiscord通知内のリンクまたはスプレッドシートから機器をリモートでオフにできる。

在宅時は5分ごとに室温を取得してスプレッドシートに記録し、設定した目標温度帯（デフォルト22〜23℃）を外れた場合にクーラーを自動で稼働・調整する。クーラー稼働時や設定変更時はDiscordに通知する。

### 想定する利用者の特性
自宅でテレビ・照明・クーラー等を利用する人全般が対象。特に、外出後に機器の消し忘れが不安な人、節電・省エネを意識している人、快適な室温を手動管理する手間を省きたい人にとって有益なシステムである。また、高齢者や子供のみが自宅にいる場合に、家族が遠隔で室内環境を把握・操作できるという用途でも有用である。

## 詳細

### 機能要求

**［外出時：消し忘れ通知機能］**

- ユーザーは、自宅の位置（緯度・経度）・ジオフェンス半径・GPS取得間隔・監視対象機器の有効無効をスプレッドシートの所定のセルで設定できること
- ユーザーが自宅から設定した距離を2回連続で超えた場合に、システムが外出を自動検知できること
- 外出検知時に、監視対象機器の稼働状況が自動取得されること
- いずれかの機器がオン状態であった場合、オン状態の機器名・稼働継続時間・設定内容を含む通知がDiscordの指定チャンネルに送信されること
- ユーザーはDiscord通知内のリンクまたはスプレッドシートから、該当機器をリモートでオフにできること
- 機器をオフにした際は、操作日時と対象機器名がスプレッドシートに記録されること
- 帰宅（ジオフェンス内に2回連続で再検知）した時点で外出ステータスがリセットされること
- 外出中も5分ごとに室温・湿度を取得し、スプレッドシートに記録できること <!-- 追加 -->
- 室温が設定した上限（デフォルト：35℃）を超えた場合、Discordの指定チャンネルへ「室内高温アラート」が送信されること <!-- 追加 -->
- 室内高温アラートのしきい値（室温上限）をスプレッドシートの所定のセルで設定できること（デフォルト：35℃） <!-- 追加 -->

**［在宅時：最適温度維持機能］**

- ユーザーは、目標温度帯（最低温度・最高温度）をスプレッドシートの所定のセルで設定できること（デフォルト：22〜23℃）
- 在宅時に5分ごとに室温・湿度・不快指数をスプレッドシートに記録できること
- 室温が目標温度帯の上限を超えた場合、クーラーが自動で稼働すること
- 室温が目標温度帯の下限を下回った場合、稼働中のクーラーが自動で停止すること
- クーラーの稼働・停止時にDiscordの指定チャンネルへ通知が送信されること
- 定時通知の送信時間帯（開始時刻・終了時刻）をスプレッドシートの所定のセルで設定できること（デフォルト：7:00〜23:00） <!-- 変更 -->
- 定時通知の送信間隔（30分 / 1時間 / 2時間）をスプレッドシートの所定のセルで設定できること（デフォルト：1時間） <!-- 変更 -->
- 湿度が除湿モード切り替えしきい値以上、かつ室温が目標温度帯の上限未満の場合、冷房ではなく除湿（ドライ）モードで運転できること <!-- 追加 -->
- 除湿（ドライ）モードへの切り替え時は、冷房稼働時と同様にDiscordの指定チャンネルへ通知が送信されること <!-- 追加 -->
- 除湿モードの稼働しきい値（湿度）をスプレッドシートの所定のセルで設定できること（デフォルト：70%） <!-- 追加 -->

**［節約機能：電気代アラート機能］**

- ユーザーは、電気代の月次上限額をスプレッドシートの所定のセルで設定できること <!-- 変更 -->
- ユーザーは、電気代の日次・週次上限額をスプレッドシートの所定のセルで設定できること <!-- 追加 -->
- 日次・週次・月次のいずれかの上限額を超過した場合、Discordの指定チャンネルへアラートが送信されること <!-- 変更 -->

### 非機能要求

- 外出検知から消し忘れ通知の送信まで、5分以内に完了すること
- GPS取得間隔はデフォルトで2分ごととし、ユーザーが変更可能であること
- 室温が目標温度帯を逸脱してからクーラー操作完了まで、3分以内であること
- Discord通知はDiscord Webhook URLを使用し、URLをスプレッドシートの所定のセルに入力することで設定できること
- Nature Remo API または Discord Webhook の呼び出しが失敗した場合、1回リトライを実行し、それでも失敗した場合はDiscordの指定チャンネルへエラー通知を送信すること <!-- 追加 -->
- APIエラー発生時は、エラー発生日時・対象API・エラー内容をスプレッドシートに記録すること <!-- 追加 -->

## 各モジュールの詳細仕様

---

### main.js

**役割：** トリガーから呼び出されるエントリーポイント。全モジュールを順番に呼び出す。

**GASトリガー設定：**
- 関数名：`runEvery5Minutes`
- 種類：時間ベースのトリガー → 5分おき

**処理フロー：**

```
runEvery5Minutes()
  ├── 1. sheet.js: 設定値を読み込む
  ├── 2. remo.js: センサ値を取得（温度・湿度・照度）
  ├── 3. sensordata.js: 不快指数を算出
  ├── 4. sheet.js: センサログに記録
  ├── 5. aircon.js: 空調制御の要否を判定・実行
  │     ├── 冷房/除湿/停止の命令を remo.js に渡す
  │     └── 操作した場合 → sheet.js に操作ログ記録
  │                      → discord.js に操作通知を送信
  └── 6. 定時通知の判定（現在時刻・前回通知時刻と比較）
        └── 送信条件を満たす場合 → discord.js に定時報告を送信
```

---

### sheet.js

**役割：** スプレッドシートの設定値読み込み・各ログ書き込みを一元管理する。

**主要関数：**

```javascript
// 設定値をオブジェクトで返す
getSettings()
// 戻り値例：
// {
//   tempMin: 22,
//   tempMax: 23,
//   humidityThreshold: 70,
//   notifyStart: 7,
//   notifyEnd: 23,
//   notifyInterval: 60,
//   remoToken: "xxxxxx",
//   discordUrl: "https://discord.com/api/webhooks/...",
//   deviceName: "エアコン"
// }

// センサログに1行追記する
appendSensorLog(timestamp, temp, humidity, illuminance, discomfortIndex, acState)

// 操作ログに1行追記する
appendOperationLog(timestamp, operation, temp, humidity, di, result, errorDetail)

// 最終通知時刻を取得する（定時通知の間隔管理用）
getLastNotifiedTime()

// 最終通知時刻を更新する
setLastNotifiedTime(timestamp)
```

---

### remo.js

**役割：** Nature Remo 3 の REST API を呼び出し、センサ値の取得とエアコン制御を行う。

**使用エンドポイント：**

```
GET  https://api.nature.global/1/devices
     → 全デバイスのセンサ値を取得

POST https://api.nature.global/1/appliances/{applianceId}/aircon_settings
     → エアコンの設定を変更（モード・温度・電源）
```

**認証：** リクエストヘッダーに `Authorization: Bearer {APIトークン}` を付与

**主要関数：**

```javascript
// センサ値を取得して返す
getSensorData(token, deviceName)
// 戻り値例：{ temp: 26.5, humidity: 72, illuminance: 340 }

// エアコンを指定モードで操作する
// mode: "cool"（冷房）/ "dry"（除湿）/ "off"（停止）
// temp: 設定温度（停止時は不要）
controlAircon(token, applianceId, mode, temp)
// 戻り値：{ success: true } または { success: false, error: "..." }
```

**エラー処理：**
- HTTPレスポンスが200以外の場合は1回リトライ
- 2回失敗した場合は `{ success: false, error: エラー内容 }` を返す

---

### sensordata.js

**役割：** センサ値から不快指数を算出する。

**不快指数の計算式：**

```
DI = 0.81 × T + 0.01 × H × (0.99 × T − 14.3) + 46.3
T：気温（℃）
H：湿度（%）
```

**主要関数：**

```javascript
// 不快指数を算出して返す（小数第1位で四捨五入）
calcDiscomfortIndex(temp, humidity)
// 例：calcDiscomfortIndex(26, 72) → 77.3
```

---

### aircon.js

**役割：** センサ値と設定値を比較し、空調の操作要否・モードを判定して実行する。

**制御判定ロジック（優先順位順）：**

```
1. 室温 > tempMax かつ エアコンOFF
   → 冷房モードで起動（設定温度：tempMax）

2. 湿度 >= humidityThreshold かつ 室温 <= tempMax かつ エアコンがOFFまたは冷房中
   → 除湿モードに切り替え（設定温度：除湿モードのため不要）

3. 室温 < tempMin かつ エアコンが冷房または除湿で稼働中
   → エアコン停止

4. 上記いずれにも該当しない
   → 操作なし
```

**主要関数：**

```javascript
// 判定と実行を行い、操作結果を返す
// 操作なしの場合は null を返す
evaluateAndControl(settings, sensorData, currentAcState)
// 戻り値例（冷房開始時）：
// { operation: "冷房開始", temp: 26.5, humidity: 72, di: 77.3, result: "成功" }
```

**エアコンの現在状態管理：**
- GASはステートレスなため、スプレッドシートのセンサログ最終行の「エアコン状態」列から現在状態を取得する

---

### discord.js

**役割：** Discord Webhook APIにメッセージをPOSTする。

**使用エンドポイント：**

```
POST {Discord Webhook URL}
Content-Type: application/json
Body: { "content": "メッセージ本文" }
```

**主要関数：**

```javascript
// 空調操作時の通知
sendAirconNotification(webhookUrl, operation, temp, humidity, di)
// 送信例：
// 【冷房開始】2025/04/22 14:05
// 室温：26.5℃ / 湿度：72% / 不快指数：77.3
// 設定温度：23℃

// 定時報告通知
sendScheduledReport(webhookUrl, temp, humidity, di, acState)
// 送信例：
// 【定時報告】2025/04/22 14:00
// 室温：26.5℃ / 湿度：72% / 不快指数：77.3
// エアコン：冷房稼働中

// APIエラー通知
sendErrorNotification(webhookUrl, targetApi, errorDetail)
// 送信例：
// 【エラー】Nature Remo API 呼び出し失敗
// 発生時刻：2025/04/22 14:05
// 詳細：HTTPステータス 503
```

## GASトリガーの設定手順

GASエディタ（またはclaspでpush後）に以下のトリガーを設定する。

| 関数名 | トリガーの種類 | 実行間隔 |
|---|---|---|
| `runEvery5Minutes` | 時間ベース | 5分おき |

> **注意：** GASの時間トリガーは±数分のズレが発生します。  
> 「毎時0分ちょうど」の厳密な実行は保証されないため、  
> 定時通知の判定は「現在時刻が設定間隔の倍数分を過ぎているか」と  
> 「前回通知からの経過時間」の両方で判定するロジックにします。

## Phase 2 移行時に追加するモジュール

Phase 1完成後に以下を追加する。

| ファイル名 | 役割 |
|---|---|
| `gps.js` | IFTTT/Shortcutsからの座標受信・ジオフェンス判定 |
| `device_monitor.js` | 外出時の機器稼働状態チェック・消し忘れ検知 |
| `electricity.js` | 電気代試算・アラート送信 |

Phase 2では `main.js` に `doPost(e)` 関数（WebhookエンドポイントとしてIFTTTから呼ばれる）を追加し、在宅/外出ステータスの管理をスプレッドシートに持たせる。
